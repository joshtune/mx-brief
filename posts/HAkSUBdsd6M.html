Adversarial Dev: Why Pitting Two Coding Agents Against Each Other Produces Far Better Code

TL;DR

Cole Medin argues that sycophancy — AI models agreeing with themselves and the user too readily — is the biggest reliability problem in AI coding today, and it's getting worse as models improve. His proposed solution: a GAN-inspired multi-agent harness where a separate evaluator agent adversarially critiques a generator agent's code, negotiating evaluation contracts and iterating through sprints automatically. He built and open-sourced harnesses for both Claude Code and Codex to prove the approach works.

The Core Problem: Sycophancy in AI Coding

Medin opens with what he considers the biggest problem in AI right now: sycophancy. While sycophancy — the tendency of AI models to agree with and reinforce user opinions — is discussed broadly in the LLM space, Medin argues it hits hardest in AI coding. His central metaphor: having a coding agent evaluate its own work is like a student grading their own homework. The agent will point out a couple of superficial flaws to make the review seem legitimate, but the deeper problems get swept under the rug.

Critically, he claims sycophancy appears to be getting worse as LLMs become more powerful, not better. The bias builds up throughout a conversation where you've injected your opinions and the agent has generated code based on them — the agent is always going to give itself a gold star when asked to review that same work.

The Solution: GAN-Inspired Adversarial Dev Architecture

The solution Medin proposes is giving your coding agent a 'sparring partner' — a completely separate coding agent with its own context session whose sole job is to rip apart the main agent's implementation. He calls this 'adversarial dev' because the second agent is designed to be nitpicky, play devil's advocate, and question even deeper architectural decisions.

The approach draws direct inspiration from Generative Adversarial Networks (GANs) in machine learning. In a GAN, one model generates images while a discriminator model tries to distinguish generated images from real ones. The generator improves by trying to trick the discriminator. Medin extrapolates this into coding: the generator agent writes code, the evaluator agent critiques it, and the generator improves by trying to satisfy (or 'trick') the evaluator. As long as the evaluation criteria actually point toward a working application, this adversarial dynamic produces better results.

Medin credits Anthropic's research as the primary inspiration. Anthropic published a blog post about their evolving long-running coding agent harness that incorporated this adversarial approach. However, Medin notes that unlike previous Anthropic blog posts, they did not open-source the implementation this time. So he built everything from the ground up himself.

How the Harness Works: Three Agents, Sprint Cycles, and Negotiated Contracts

The full system involves three agents, not two:

1. **Planner Agent** — Takes your initial user prompt and expands it into a comprehensive product specification. The planner's system prompt begins with 'You are a product architect. Your job is to take a brief user description and produce a comprehensive product specification.' You don't need to be overly specific in your prompt because the planner handles expanding it into a full spec optimized for the harness.

2. **Generator Agent** — The implementer/software engineer that writes the actual code.

3. **Evaluator Agent** — Described as 'a skeptical QA engineer' in its system prompt. Its sole job is to adversarially evaluate the generator's output.

The most interesting part of the system is what happens before any code is written. The generator and evaluator first negotiate a contract — they agree upfront on how to break the spec into manageable sprints/phases and how each phase will be evaluated. This includes specific criteria with scores from 1 to 10 and threshold values that must be met or exceeded. Medin describes this as establishing 'etiquette' — yes, it's adversarial, but they're agreeing on what the battle looks like before it begins.

The generator then implements each sprint, the evaluator scores it against the agreed criteria, and if any criterion falls below its threshold, the generator retries — up to a maximum of three attempts per sprint. Once all criteria pass, they move to the next sprint. This continues until the full spec is implemented.

Repository Structure and Implementation

Medin built and open-sourced harnesses for both Claude Code and Codex, deliberately structured the same way to demonstrate the approach isn't tied to Anthropic's tooling. The repository contains:

- A `planner` module for both Claude and Codex versions

- A `generator` module for both

- An `evaluator` module for both

- A `harness` that ties everything together (harness.ts)

- A `shared` folder containing all the prompts for different stages

- An example `prompt.md` as a starting point

He notes you can even mix providers — for example, having Claude be the implementer and Codex be the evaluator. The harness is kicked off with a single bun command, and he suggests using your coding agent to help create the initial prompt.md and even to read the README and run the harness command for you.

Evidence: What This Actually Produces

Medin presents two concrete examples of what the harness can build:

**Anthropic's Example — Retro Game Maker:** Anthropic tested both a single agent and the adversarial harness on building a retro game maker where users could define sprites and play the game. The single-agent version looked decent visually but gameplay was completely broken — nothing worked. The harness version (after the evaluator pushed back on the generator's implementation) produced a working sprite editor, level designer, and functional gameplay.

**Medin's Example — RAG YouTube Chat App:** Medin built a full-stack RAG application that ingests his YouTube content, runs a RAG pipeline on the backend, and presents a polished chat UI on the frontend. The app featured working LLM-powered Q&A, token streaming, source citations, and what he described as a premium feel comparable to ChatGPT or Claude.ai. He one-shotted this — sent in a single request and let the harness work through it over approximately 4 hours. He states explicitly that a single coding agent session would not have been able to create something this elaborate.

The kicker: he built the RAG app using Claude Sonnet 4.6, not Opus 4.6. He states that while Opus 4.6 is 'way better' than Sonnet 4.6 based on his extensive use of both, Opus would not have been able to one-shot this application without the harness — but Sonnet did with the harness. This is his key evidence that the harness architecture matters more than raw model capability.

Using Your Subscription, Not API Credits

Medin addresses the question of cost for running long harnesses. He references recent controversy about whether Anthropic and Codex subscriptions can be used with the Agents SDK or headless mode. He cites a clarification from a member of the Anthropic team (referred to as 'the is one of the members of the Anthropic team' — the name is unclear in the transcript) stating that using your subscription with the Agents SDK or headless Claude mode is allowed as long as it's for local development and experimentation. If you're building a business where other people would use your subscription, that violates the terms of service and you should use API keys instead.

This means for local experimentation with the harness, you don't need to pay for API credits separately — you can authenticate with your existing subscription for either Claude Code or Codex.

Downsides and Trade-offs

Medin acknowledges the primary downside: token cost and time. The full speccing process with the loop of implementation, evaluation, and iteration runs for a long time — his RAG app example took approximately 4 hours. He explicitly notes 'coding agents are never going to be perfect' and that the harness won't produce something immediately production-ready.

However, he counters this with two arguments. First, the harness enables using faster and cheaper models (like Sonnet instead of Opus) while still getting strong results, which can actually save tokens overall compared to repeatedly prompting a more expensive model. Second, the reliability improvement is worth the overhead — setting up a system where feedback and iteration are handled automatically means you don't have to personally guide each sprint and work through AI slop yourself. He positions the harness as ideal for proofs of concept and getting started on projects quickly, not necessarily for production-ready code in one shot.

Key Takeaways

  • Never have a coding agent evaluate its own work in the same context session — sycophancy means it will overlook its own deeper mistakes, and this problem is getting worse with more powerful models
  • A GAN-inspired adversarial architecture (separate generator and evaluator agents with independent context) produces dramatically more reliable results than single-agent coding sessions
  • The contract negotiation step — where generator and evaluator agree on sprint breakdown and scoring criteria before any code is written — is critical to making the adversarial dynamic productive rather than chaotic
  • The harness architecture can matter more than raw model power: Claude Sonnet 4.6 with the adversarial harness outperformed what Opus 4.6 could do without it on a complex full-stack app
  • Use these harnesses for rapid proof-of-concept development and project bootstrapping, not necessarily for production-ready code — but expect this multi-agent approach to become standard for AI-assisted development
