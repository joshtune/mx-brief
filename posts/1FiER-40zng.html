Build Your Own AI Second Brain with Claude Code and Obsidian

TL;DR

Cole Medin walks through the full architecture of his Claude Code + Obsidian-based AI second brain that he claims saves him 12+ hours per week. Rather than using an off-the-shelf solution like Open Claw, he argues for building your own for security and customization reasons, and provides a GitHub template that generates a phased PRD to guide you through building it.

Why Build Your Own Instead of Using Open Claw?

Medin opens by addressing why you'd build a custom second brain rather than running an existing solution like Open Claw. He introduces the concept of the "lethal trifecta" — three criteria that, when all present, make an AI agent highly susceptible to security risks like prompt injection:

1. **Private data access** — Your agent has access to email, calendar, or other sensitive information. This always applies to a second brain, otherwise it wouldn't be useful.

2. **Untrusted content** — Any input that doesn't come from you (incoming emails, web page content) could theoretically contain malicious prompt injection instructions.

3. **Exfiltration vector** — The agent can send data outward (messages, API calls), giving a potential attacker a way to extract your private data.

He argues that if only two of the three are present, the risk drops significantly. But a useful second brain inherently has all three. The key insight: since you can't avoid the trifecta, you need to limit each pillar as much as possible. Open Claw is a massive codebase you don't fully understand, giving broad control to an agent without you knowing exactly what permissions are in play. By building your own, you define permissions upfront, start with zero-trust, and layer on capabilities incrementally.

However, he doesn't advocate ignoring Open Claw entirely. Instead, he pointed Claude Code at the Open Claw GitHub repository (which is fully open source) and told it to take inspiration from specific parts — like the memory layer implementation with soul.md/user.md, and the heartbeat for proactive agent behavior. The approach is: build it yourself, but stand on the shoulders of giants.

Architecture Overview: The Memory Layer

The memory layer is the core of the second brain, directly inspired by Open Claw's approach. It consists of markdown files managed inside an Obsidian vault:

- **soul.md** — The personality definition for the second brain. Taken directly from Open Claw's concept.

- **user.md** — Information about you that evolves over time as the agent learns your preferences, communication style, and work patterns.

- **memory.md** — Key decisions, lessons learned, and important facts. This is described as "probably the most important file for the entire second brain" since it's always loaded into context.

These core files are loaded into Claude Code's context automatically through a **session start hook**. This means every conversation begins with the agent already knowing your personality, preferences, and recent decisions.

Obsidian serves as the canvas — you can view and edit the generated documents, diagrams, and memory files visually. Every document visible in the Obsidian graph is part of the second brain's knowledge base.

Context Persistence: Hooks and Daily Logs

Hooks are a critical part of the architecture beyond just loading memory at session start. Medin describes three key hooks:

1. **Session start hook** — Automatically loads soul.md, user.md, and memory.md into the Claude Code context at the beginning of every conversation.

2. **Pre-compact hook** — When a conversation gets very long and memory compaction is triggered, this hook saves the conversation to a daily log file before compacting. This ensures nothing is lost.

3. **End session hook** — When a Claude Code session ends, this hook also saves the conversation to the daily log.

The daily log is a raw dump of everything from every conversation on a given day. Medin showed that he has a file for every single day since he started running his second brain.

A **daily reflection process** then runs once a day via a cron job. This uses the Claude Agent SDK under the hood (not manual prompting) to perform a "promotion process" — it goes through the lengthy raw daily log and extracts key decisions, lessons learned, and important facts, promoting them into the memory.md file. This promotion process is described as "one of the most important parts of the system that allows the agent to evolve over time."

Memory Search via RAG

Since memory.md is a concise distillation, it won't always contain everything you might want to reference later. To handle this, the agent can perform RAG (retrieval-augmented generation) over all daily logs.

All daily logs are indexed into a **SQLite database** (Medin noted Postgres would also work). The agent can then search through the full history of daily logs to find specific information that wasn't important enough to be promoted to memory.md but is still relevant to a current conversation. The searchable corpus includes everything visible in the Obsidian graph view.

Skills: Giving Arms to Your Second Brain

Capabilities are added through Claude Code skills — no MCP servers needed. Medin emphasized this point: "We don't need other applications. We don't even need MCP servers. I literally use my skills for everything."

His .claude/skills directory contains many skills he uses at least weekly, including:

- **Excalidraw diagram generation** — Creates diagrams like the architecture diagram shown in the video

- **PowerPoint generation**

- **YouTube script creation** — Uses the memory layer to understand his tone of voice and past scripting patterns

- **Image/thumbnail generation**

- **Habit tracking**

- **A core integrations skill** — Explains all direct integrations and exactly how to use the APIs set up for each service

The critical insight is that skills become personalized through the memory layer. A diagram skill would produce generic output without the memory context — but because soul.md, user.md, and memory.md are always loaded, every skill's output is tailored to the user.

Direct Integrations via Python API Layer

For connecting to external services, Medin built a Python API layer with carefully controlled permissions for each integration:

- **Gmail** — Can create drafts, but cannot send emails

- **Slack** — Can read messages, but cannot post messages

- **Google Calendar** — Can read events, but cannot create or manage them

- **Google Docs** — Cannot create documents

- **Asana** — Can manage tasks, but only in certain projects

- **Circle** (Dynamis AI community platform) — Has access

The philosophy is **zero-trust by default**: start with read-only access, then add write capabilities incrementally and deliberately. Because it's a custom build, you can be "very nitpicky about the permissions."

All integrations follow a common pattern, which has a powerful compound effect: the second brain can reference existing integrations and its memory of how it built things in the past to one-shot new integrations with Claude Code. The same pattern was used to build the Slack chat interface, and it could be easily replicated for Telegram, Discord, or any other platform.

The Heartbeat: Proactive Agent Behavior

The heartbeat is what makes the second brain proactive rather than purely reactive. It follows this flow:

1. **Deterministic data gathering** — Before Claude Code is even invoked, the system gathers context from all integrated APIs (Gmail, task management, GitHub, etc.). This is the deterministic phase — no LLM involved yet.

2. **Claude Agent SDK invocation** — All gathered information is sent into Claude Code programmatically via the Agent SDK with a preconfigured prompt. The agent is not being prompted manually by the user.

3. **Reasoning and action** — The agent reasons about what needs attention: drafting email replies, handling GitHub pull requests, determining what to help with next.

4. **Slack notification** — The agent sends a summary notification to Slack, keeping the user in the loop on everything it did proactively.

5. **Threaded follow-up** — The user can reply to any notification in a Slack thread to continue the conversation on that specific item.

Medin describes the heartbeat as "the thing that saves me the most time out of everything in the second brain."

The Starter Template: Generating Your PRD

Rather than releasing his full codebase (which he says would just create "another Open Claw"), Medin provides a GitHub repo (linked in the video description) with a template system for generating a phased PRD (Product Requirements Document) for your own second brain.

The workflow is:

1. **Clone the repository** from the link in the video description.

2. **Find and edit the `my-second-brain-requirements.md` file.** Fill in your personal information (name, timezone), check off which platforms you want to integrate, list tasks you want automated, and set your preferences.

3. **Set your proactivity level** — Options range from "observer" (read-only) to "partner" (fully autonomous, can send emails on your behalf). Medin uses "advisor" level.

4. **Define security boundaries** — Specify what the agent should never do.

5. **Configure memory categories** — What types of knowledge should it track.

6. **Choose infrastructure** — Local-only on your computer, or hosted remotely (e.g., Digital Ocean, Hostinger droplet) for 24/7 Slack access.

7. **Set integration priorities** — This determines the phase ordering in the PRD.

8. **Run the Claude Code skill** to generate the PRD. The skill does extensive research and accounts for all your customizations.

The repo also includes a pre-filled example requirements file you can use as a reference. You can also point the skill at external repos like Open Claw for additional inspiration.

/create-second-brain-prd <path-to-your-filled-requirements.md>

Building in Phases

The generated PRD outputs to a `.agents/plans/` folder and breaks the build into sequential phases that map to the architecture components:

1. **Phase 1: Memory foundation** — Set up soul.md, user.md, memory.md in your Obsidian vault. Already useful as a partial second brain.

2. **Phase 2: Hooks for context persistence** — Build session start, pre-compact, and end session hooks to load memory and save daily logs.

3. **Phase 3: Memory search** — Index daily logs into SQLite and implement RAG capability.

4. **Phase 4+: Integrations, skills, and capabilities** — Add direct integrations, build skills, configure the heartbeat.

The explicit advice is to NOT have your coding agent try to build the entire second brain in one shot. Build one phase at a time by opening a new Claude Code session, pointing it at the PRD, and saying something like "Let's build phase one" or "Let's build phase four." Kick off a planning session for each phase, tweak anything from the PRD, write the code, test it, and move on.

Medin claims you can have something as powerful as Open Claw in "just a couple of days" with this approach, but it will be entirely your own, simpler to maintain, and more secure.

Key Takeaways

  • The 'lethal trifecta' (private data access + untrusted content + exfiltration vector) is unavoidable in a useful second brain — so limit each pillar by building your own with controlled permissions rather than running a large codebase you don't understand.
  • The memory layer (soul.md, user.md, memory.md) loaded via session start hooks is the core of the system — everything else builds on top of it.
  • Use a daily log + promotion process (via cron job and Claude Agent SDK) to let your second brain evolve its memory over time without manual curation.
  • Start with zero-trust permissions for all integrations (read-only), then add write capabilities incrementally and deliberately.
  • Claude Code skills plus the memory layer is all you need for capabilities — no MCP servers or additional applications required.
  • The heartbeat (proactive agent via Agent SDK + cron) is the highest-ROI component — it gathers context from APIs, reasons about what needs attention, acts, and notifies you via Slack.
  • Build in phases, not all at once. Use the provided template to generate a phased PRD, then build one phase per Claude Code session.
  • Take inspiration from existing open-source projects like Open Claw by pointing your coding agent at their repos, but maintain your own simpler, more secure implementation.
