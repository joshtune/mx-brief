What Claude Code's Leaked Source Code Reveals: 8 Insights for Power Users

TL;DR

Claude Code's source code was inadvertently leaked via an NPM package source map, revealing that it's a full agent runtime — not just a chatbot in your terminal. The creator breaks down 8 practical insights from the leak, covering hidden commands, the memory system, permissions tuning, multi-agent architecture, and a framework for using Claude Code like a top 1% user.

How the Leak Happened

Anthropic published an NPM package that included a source map file pointing to readable TypeScript source on their servers. A security researcher found it, followed the breadcrumbs, and the full source — roughly 2,000 files and over 500,000 lines of production code — was mirrored publicly on GitHub within hours. The creator notes a disclaimer: he does not show any actual source code in the video and cautions viewers about downloading it, since Anthropic still owns the copyright and has issued DMCA takedown notices.

A Fun Detail: 187 Spinner Verbs

As a quick appetizer before the main insights, the creator points out that the loading spinner messages you see while Claude Code is thinking come from a file containing 187 spinner verbs. These range from standard ones like "computing" and "generating" to humorous ones like "boondoggling," "discombobulating," "fibridding," and "moonwalking." The creator's take: "That's the energy I want from the people building my tools."

Insight 1: It's Not What You Think It Is

The first and foundational insight is that Claude Code is not simply "Claude but in your terminal" — it's a full agent runtime. The source code reveals it's built with Bun, TypeScript, and React, and includes a tool system, a command system, a memory system, a permission engine, a task manager, a multi-agent coordinator, and an MCP client and server, all wired together under one execution pipeline.

The execution flow works like this: your input hits a CLI parser, goes to something called the "query engine," calls the LLM API, runs a tool execution loop, and then renders results back in your terminal.

The creator's argument: if you're using Claude Code like a chatbot — just typing questions and hoping for good answers — you're using maybe 10% of what it can actually do. The rest of the value is in the systems built around the model.

Insight 2: The Command Service You're Ignoring

The source code reveals roughly 85 slash commands in Claude Code. The creator argues most users know maybe five of them — typically just /help and maybe /context. Power users, he says, use these commands like shortcuts and get 10x more value.

The commands he highlights as most important:

- **/init** — Sets up project context by generating a claude.md file that acts as Claude Code's operating manual for your repository.

- **/plan** — Puts Claude Code into planning mode. Instead of immediately executing, it maps out the full approach first and asks before touching files. The creator calls this "huge" for complex work and notes it saves tokens long-term.

- **/compact** — Compresses conversation history to keep important context but drop noise. The creator calls this "one of the fastest fixes" for token burn. You can also pass a prompt to it (e.g., "/compact keep all important information about the website integration") to control what gets preserved.

- **/review and /security review** — Built-in structured code review workflows. The creator notes that having these as dedicated commands signals that review is a "first-class workflow" in the product, not an afterthought.

- **/context** — Manages what files Claude Code is paying attention to. Important because every file in context costs tokens.

- **/cost** — Shows what you've actually spent in a session. The creator says most people have no idea how much a session costs until they see the bill.

- **/resume and /summary** — Let you pick up where you left off between sessions without re-explaining everything.

The creator's key point: better prompts are one lever, but knowing these commands is a completely separate lever that most people aren't touching.

Insight 3: The Memory System Is Way More Important Than You Think

The source code reveals a full memory system, centered on a file called claude.md. The creator argues that most people either ignore this file or dump random notes into it, and that's a huge mistake.

His framing: claude.md is not documentation — it's operating context. If Claude Code is an employee, then claude.md is their onboarding document. It tells them: here's how we do things, here's what matters, here's what we never do, here's how this project is structured.

The best users, he says, keep claude.md short, opinionated, and operational. He gives specific examples of what belongs in it: "We use TypeScript strict mode — always." "Tests go in test folders next to the source file." "Never modify the database schema without running migrations." "Use pnpm, not npm." The content should be decision rules, constraints, and conventions — not a novel about your project's history.

The source also reveals multiple layers to the memory system: user-level memory, extracted memories, and even team memory synchronization. This means Claude Code has persistent memory mechanisms across project, user, and session contexts. The creator emphasizes that claude.md is "one of the highest-leverage inputs" shaping behavior in future sessions, and that these files get injected before every single chat.

His strongest recommendation: "If you get nothing else from this video, go update your claude.md file today — whether that's on the global level or the project level."

Insight 4: Permissions Are Why Claude Might Feel Slow

The source code reveals a deep permission system with multiple modes: a default mode where it asks about everything, a plan mode, and bypass/auto modes where it can just execute.

The creator identifies the "real gem" as wildcard permissions — the ability to set rules like "allow all git commands" or "allow all file edits in my src folder." Instead of Claude Code asking 15 times whether it can run git status, edit a file, or run a test, you set the rule once and it just works.

These are configured in settings.local.json or settings.json, where you can set global, user-level, or project-level permissions for actions you always want to allow, always deny, or always want it to ask about.

The creator frames this as one of the highest-ROI changes for recurring workflows: "This lets you go from babysitting every action to actually being able to let it operate like an agent while you step away."

Insight 5: It's Built for Multi-Agent Work

The source code reveals a full coordinator subsystem, agent tools, team tools, and a task system designed for background and parallel work. The creator says this one surprised him.

In plain terms: the architecture supports decomposition — splitting work across multiple agents that can run in parallel. The creator paints a picture: one agent exploring your codebase, another implementing changes, another validating tests. There are even concepts for background tasks — work that continues while you're focused on something else.

The creator acknowledges a caveat: how much of this is fully user-facing today versus still rolling out is harder to say from the source alone. But the architectural intent is clear — Claude Code is designed to handle complex multi-step work by breaking it apart, not by cramming it all into one thread.

Practical takeaway: instead of one massive prompt like "refactor this entire module, update the tests, and fix the documentation," break that up. Let Claude Code decompose the work. The best results, he argues, come from breaking complex work into clear sequential or parallel steps rather than hoping one giant instruction to one giant agent gets everything right.

Insight 6: MCP, Plugins, and Skills Are the Real Extension Layer

The source code confirms that MCP (Model Context Protocol) isn't just supported by Claude Code — it's baked into the architecture. Claude Code is both an MCP client and an MCP server. It can connect to external tools through MCP, and other systems can connect to it.

Beyond MCP, the source reveals a skills and plugin layer. Power users can build repeatable workflows, custom capabilities, and domain-specific extensions that compound over time.

The creator's argument: this is where Claude Code stops being just a coding tool and becomes an integration layer. You can connect it to databases, APIs, internal tools, documentation systems — anything with an MCP server — and then layer skills and plugins on top for repetitive tasks. The more systems you connect, the more useful it becomes.

Insight 7: Features We Don't Have Access to Yet

The codebase includes checks for something called "user type," and one of the values is "ant" — meaning Anthropic. This indicates certain capabilities are gated behind internal feature flags.

The source references specific unreleased features: voice mode, a system called "Chyros," a daemon mode, and a coordinator mode — all behind flags suggesting they're internal, experimental, or being rolled out gradually.

The creator is transparent about the limits of what can be concluded: "We can't say for sure how different the internal experience actually is. Some of these could be early prototypes. Some might be close to shipping publicly. The source alone doesn't tell us that."

What the source does confirm is that Claude Code is heavily feature-flagged, meaning different users may already be getting meaningfully different experiences depending on their environment, build, or rollout group. The creator points to how many features Claude Code has been shipping recently as evidence that they've been built and tested internally before release, noting he's heard rumors that computer use had been around internally for a long time before its public release.

Insight 8: How to Actually Use This — The 7 Habits of Top 1% Users

The creator ties everything together with what he calls the single biggest insight from the entire source code: "Top users don't just write better prompts. They design a better operating environment for their Claude Code."

He lays out seven specific practices:

**1. Treat claude.md like a force multiplier.** Keep it short, opinionated, updated regularly. Route to other files. This single file shapes every interaction.

**2. Learn the command service.** You don't need all 85 commands, but /plan, /compact, /context, /review, /cost, and /resume will change your daily workflow immediately.

**3. Configure permissions for recurring workflows.** Stop babysitting every action. Set wildcard rules for stuff you do every day.

**4. Think in decomposition, not monolithic prompts.** Break complex work into a search phase, a plan phase, an execute phase, and a verify phase. Let Claude Code use its architecture the way it was designed.

**5. Manage context like it's money — because it literally is.** Use /compact when conversations get long. Use /context to control what's loaded. Use /summary and /resume to carry work across sessions. Every unnecessary file in context is tokens you're paying for. The creator explicitly calls context management "an actual discipline" and "something you need to build a habit around."

**6. Connect things to it.** MCP servers, CLIs, plugins, skills — the more tools Claude Code can access, the more valuable it becomes.

**7. Treat it like infrastructure, not an app.** The source reveals a massive configuration surface: model routing, sub-agent model overrides, shell behavior, privacy controls, the ability to route through different cloud backends like AWS Bedrock or Google Vertex. Most people never touch these settings, but there's real leverage in tuning the environment to fit your workflow.

  • Break complex work into phases (search, plan, execute, verify) rather than sending monolithic prompts — the architecture is designed for decomposition and parallel agents.
  • Manage context like money: use /compact to compress history, /context to control loaded files, and /summary + /resume to carry work across sessions without token waste.
  • Connect Claude Code to external systems via MCP servers and build repeatable workflows with skills and plugins — the more you connect, the more useful it becomes.
  • The codebase is heavily feature-flagged with unreleased capabilities (voice mode, daemon mode, coordinator mode) — stay current on updates to leverage new features as they ship.
