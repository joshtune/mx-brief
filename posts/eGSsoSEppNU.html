The Axios Supply Chain Attack: How a Single Dependency Hijack Deploys a RAT in 1.1 Seconds

TL;DR

The Axios npm package (100M+ weekly downloads) was hijacked via a compromised maintainer access token. The attacker added a single malicious dependency to package.json that installs a remote access Trojan in 1.1 seconds, then erases all evidence of itself. Affected versions are 1.14.1 and 0.30.4 — and you likely have Axios installed transitively without knowing it.

The Scope of the Attack

Axios, described as the most popular HTTP library with over 100 million downloads a week, was hijacked. NetworkChuck calls this potentially the most sophisticated and dangerous supply chain attack in history. The attacker took over a lead maintainer's account, injected a mechanism to deploy a remote access Trojan (RAT) in under 1.1 seconds, and the malware erases itself afterward — leaving no trace behind.

The critical point NetworkChuck emphasizes: you rarely install Axios directly. It's a transitive dependency — something your dependencies depend on. He states that 174,000 projects depend on Axios, and the average npm project trusts between 200 and 2,100 strangers with code execution. When you run `npm install` for any project, you're pulling in a massive tree of code written by unpaid open-source contributors.

How the Attacker Got In

The attacker compromised the account of Jason Semen (spelling uncertain from audio), a lead maintainer for Axios. Specifically, the attacker obtained a long-lived npm classic access token that gave them full publish access to the package. NetworkChuck notes that exactly how the token was obtained is not yet known — only that it was.

Once inside, the attacker changed the account email to an address NetworkChuck reads as something like "ifstoppro.mme" (exact address unclear from the audio). The Axios project, like many major open-source packages, is maintained by unpaid volunteers — NetworkChuck pulls up the commit log showing contributions from various individuals and emphasizes these are random, unpaid people maintaining critical infrastructure.

The Injection: One Line in package.json

Here's what makes this attack clever: the attacker never actually added any malicious code to Axios's source files. Instead, they added a single dependency to the package.json file — something called "plain-crypto-js" (as heard in the transcript). NetworkChuck notes that an average code reviewer would glance at this and think "Oh, that's a simple crypto thing" and move on.

This dependency was never imported into any of Axios's 86 source files. It existed solely for one purpose: to execute its postinstall script when npm installs it. The attacker also staged a clean version of this file 18 hours before the malicious one, and bypassed CI/CD pipelines by publishing directly via the npm CLI — skipping all the guardrails that would normally catch a suspicious change.

The Two Poisoned Versions

Socket.dev was credited as the first company to discover the attack. The attacker poisoned two release branches: the 1.x branch (version 1.14.1) and the 0.x branch (version 0.30.4). These two versions were poisoned within 39 minutes of each other.

NetworkChuck explains that any project using a caret range (^) for either of these releases would automatically pull the compromised version on the next `npm install`. This happens routinely in CI/CD pipelines or when developers install packages that depend on Axios.

The Kill Chain: From npm install to Full Compromise

NetworkChuck walks through the full attack chain using a diagram from Socket.dev:

1. **Postinstall script runs automatically** — no user interaction needed after `npm install`.

2. **A dropper executes**, which drops a file called `setup.js`.

3. **setup.js appears harmless** — it looks like math — but contains two layers of obfuscation using XOR and Base64 encoding, plus a phrase "order 7077" (purpose unclear from transcript). This hides the dangerous payload from static scanners.

4. **OS detection** — the deobfuscated code detects which operating system (Mac, Windows, or Linux) the machine is running.

5. **C2 contact** — it reaches out to a command and control server operated by the attacker and downloads the appropriate RAT for that OS.

6. **Cleanup** — the malware deletes `setup.js`, deletes the malicious `package.json`, and renames a pre-staged `package.md` to `package.json` (the clean version). All evidence is erased.

This entire sequence completes in 1.1 seconds after `npm install`. At that point, the attacker has full remote access to the system and the victim has no idea anything happened.

How to Check If You're Affected

NetworkChuck demonstrates checking for the compromised versions live on his machine. He starts by listing installed Axios versions. If you see version 1.14.1 or 0.30.4, he advises doing a deeper search across your entire system. He runs a multi-line search command (exact command not fully visible in the transcript) and discovers Axios in projects he custom-built himself, emphasizing how pervasive the dependency is.

He then checks whether the RAT binary actually made it onto his system using a system-level search command (he mentions having commands for Mac, Windows, and Linux in the video description). He also mentions checking for connections to the C2 server's IP address, noting that the C2 server had been taken down by the time of recording.

Remediation: What to Do If You're Compromised

NetworkChuck is emphatic: if you find evidence of the compromised versions or the RAT, do NOT just delete the files. You must treat the machine as fully compromised. His specific recommendations:

- **Rotate everything**: API keys, every credential, every token.

- **Don't treat it as a simple cleanup** — assume the attacker had full access to your system.

- He references a full remediation checklist available in the video description.

- If nothing showed up in your checks, you're "probably good for now" but should keep monitoring.

He also recommends watching John Hammond's livestream where Hammond went through all the malicious code in detail.

The Bigger Picture: Supply Chain Attacks Are Accelerating

NetworkChuck uses a coffee supply chain analogy to explain the attack's scope: poisoning one person's cup is targeted, but poisoning the water supply affects everyone — and that's the scale of compromising a package like Axios.

His broader argument is that these attacks are happening "way more often than they should" and that AI is a contributing factor — it helps developers build, but it also helps attackers execute more sophisticated attacks. He frames this not as the victim's fault: "You weren't doing anything weird or wrong. You were just installing or using software from people that you trust. It's not your fault. It's a supply chain attack."

The core tension he highlights is that critical infrastructure used by 174,000+ projects is maintained by unpaid volunteers, and a single compromised access token can cascade into a massive security incident.

Key Takeaways

  • Run `npm list axios` immediately and check for versions 1.14.1 or 0.30.4 — these are the compromised releases
  • If you find the compromised version or any RAT artifacts, treat the entire machine as compromised — rotate ALL credentials, API keys, and tokens
  • Transitive dependencies are a massive attack surface: the average npm project trusts 200-2,100 strangers with code execution on your machine
  • The attacker bypassed CI/CD pipelines by publishing directly via npm CLI — automated guardrails alone are not sufficient
  • Postinstall scripts run automatically on npm install with no user interaction — this was the execution vector
  • Socket.dev discovered the attack — consider using dependency scanning tools that can detect suspicious postinstall behaviors and new transitive dependencies
